# RideKorea Sasaki Scenario Product Plan

작성일: 2026-06-29

이 문서는 일본 라이더 “사사키” 시나리오를 기준으로 RideKorea가 어떤 서비스가 되어야 하는지, 현재 구현과의 차이, 외부 의존 없이 먼저 만들 수 있는 확장 뼈대를 정리한다.

---

## 1. 목표 경험

RideKorea는 단순한 자전거 지도 앱이 아니라, 외국인 라이더가 한국 종단을 준비하고 실제로 달리며, 먼저 달린 사람들의 기록을 참고하고, 자신의 여행기를 남기고 공유하는 플랫폼이 되어야 한다.

핵심 사용자는 다음과 같다.

- 한국 자전거 종단 정보가 부족한 외국인 라이더
- 서울-부산, 서울-목포, 제주, 금강, 낙동강 같은 장거리 라이딩을 준비하는 사람
- 공식 루트뿐 아니라 실제 라이더의 경험담, 수리소, 식당, 숙소, 교통 팁이 필요한 여행자
- 빠른 완주보다 지역을 경험하며 느긋하게 기록하고 싶은 사람

---

## 2. 사사키 시나리오 핵심 요구

### 2.1 온보딩

- 앱 첫 화면은 설명 페이지가 아니라 로그인 화면이다.
- 로그인은 Google과 Apple이 보이되, 1차 구현은 Google만 동작한다.
- 외국인 사용자에게 영어/일본어 안내가 필요하다.

### 2.2 공식 종단 루트

- 로그인 후 지도와 함께 한국 종단 루트를 선택할 수 있다.
- 서울-부산, 서울-목포, 제주, 금강 등 다양한 루트가 제공된다.
- 루트별 거리, 예상 소요 시간, 난이도, 주요 인증센터, 출발/도착 정보를 보여준다.

### 2.3 사용자 공유 루트

- 기본 공식 루트 외에 사용자들이 실제 라이딩한 루트가 공유된다.
- 공유 루트는 여행기 제목처럼 노출된다.
- 예: “베어링과 함께한 대전강변 투어”
- 공유 루트를 누르면 작성자, 이동 시간, 머문 곳, 사진, 글, 추천/댓글/공유 수를 볼 수 있다.
- 루트 위에서 출발지부터 도착지까지 작성자의 기록을 따라 읽을 수 있다.
- 사용자는 마음에 드는 공유 루트를 “내 루트”로 가져올 수 있다.

### 2.4 라이딩 중 기록

- 출발 버튼을 누르면 라이딩 기록이 시작된다.
- 위치, 속도, 시간, 거리 정보가 기록된다.
- 사진을 찍으면 지도에 자동 마커가 생긴다.
- 마커를 누르면 제목과 내용을 입력할 수 있다.
- 공식 루트를 정상적으로 지난 구간은 연한 파란색, 우회하거나 벗어난 구간은 분홍색처럼 구분할 수 있다.

### 2.5 지역 여행 정보

- 루트 중간 지역별 역사, 관광 명소, 맛집, 경치, 숙소, 수리소 정보를 볼 수 있다.
- 다른 사용자의 추천/비추천 기록을 확인할 수 있다.
- 공항에서 출발지까지 자전거를 가져가는 방법 같은 교통 팁도 필요하다.

### 2.6 지역 바우처

- 특정 지역 또는 인증 스팟에 도착하면 지역 한정 바우처가 발급될 수 있다.
- 바우처는 식당, 숙박, 수리소 등 지역 제휴처에서 쓸 수 있다.
- MVP에서는 실제 결제/정산 전까지 리워드 코드 또는 쿠폰 형태로 시작한다.

## 3. 현재 구현 상태와 차이

최신 구현 상태와 QA 기준은 별도 체크리스트에서 계속 갱신한다.

- [RideKorea Sasaki Scenario QA Checklist](./ridekorea_sasaki_scenario_qa_checklist.md)

### 이미 구현된 기반

- Google 중심 로그인 화면
- 인증 세션 Context 중앙화
- Journey / Moments / My Path / Compass 탭 구조
- 공식 코스 선택 및 지도 표시 구조
- 스팟 기반 다이어리 작성
- 공개 다이어리 Moments 피드
- My Path 내 종주 기록
- Compass 바우처/관리자 설정 골격
- 지도/GPS/종주 로직 훅 분리

### 아직 부족한 핵심

- 사용자 공유 루트 도메인
- 공유 루트 상세 화면
- 공유 루트를 내 루트로 가져오기
- 추천/댓글/공유
- 실제 주행 트랙 저장
- 공식 루트와 이탈 루트 비교
- 사진 위치 자동 마커
- 수리소/식당/숙소/교통 팁 데이터
- 일본어 포함 다국어 UX
- 지역 바우처 사용처와 실제 발급 정책

## 4. 외부 의존 작업

사용자 또는 별도 외부 준비가 필요한 작업이다.

- NAVER Map Client ID, 허용 도메인/패키지 등록
- Google OAuth 운영 Client 설정 검증
- Apple Developer 계정 및 Sign in with Apple 설정
- 한국 자전거 도로, 인증센터, 관광/교통/숙박/맛집 공공 데이터 수집
- 지역 제휴처/바우처 정책 협의
- 개인정보 처리방침, 위치정보 이용 고지, 사진 업로드 정책
- 앱스토어/플레이스토어 계정 및 배포 설정

## 5. 외부 의존 없이 먼저 할 작업 순서

### Step 1. 공유 루트 프론트 골격

목표: “피터의 루트” 같은 사용자 공유 루트를 앱 안에서 표현할 수 있게 한다.

- `SharedRoute` 타입 추가
- 샘플 공유 루트 데이터 추가
- Moments 안에 “Routes / Diaries” 보기 분리
- 공유 루트 카드 컴포넌트 추가
- “내 루트로 가져오기” 버튼은 우선 UI와 이벤트 자리만 만든다.

### Step 2. 공유 루트 상세 화면 골격

목표: 지도 + 타임라인 형태의 여행기 상세를 만들 준비를 한다.

- route id 기반 상세 화면 추가
- 작성자, 거리, 시간, 추천/댓글/공유 수 표시
- stops/timeline 표시
- 사진, 수리소, 맛집, 숙소 힌트 표시

### Step 3. 내 루트로 가져오기 내부 모델

목표: 공유 루트를 내 Journey 계획으로 복사할 수 있는 흐름을 만든다.

- 프론트 액션 플로우 설계
- 백엔드 `shared_routes` / `route_stops` / `route_reactions` 모델 설계
- Alembic 마이그레이션 준비
- API는 실제 DB 작업 전 mock 또는 skeleton endpoint로 시작 가능

### Step 4. 라이딩 트랙 기록 모델

목표: 실제 주행 경로를 저장하고 공식 루트와 비교할 준비를 한다.

- `journey_track_points` 모델 설계
- 위치 좌표, 속도, 고도, 기록 시각 저장 구조
- 추후 오프라인 큐/배터리 정책을 고려한 수집 단위 설계

### Step 5. 여행 정보 POI 모델

목표: 수리소, 식당, 숙소, 관광지, 교통 팁을 하나의 확장 가능한 구조로 받는다.

- `travel_pois` 타입 설계
- category: repair, food, lodging, scenic, transport, culture
- 추천/비추천 후기 연결 가능하도록 확장

### Step 6. 커뮤니티 반응 골격

목표: 추천, 댓글, 공유 카운트를 서비스 표면에 얹는다.

- reaction count 타입
- comment preview 타입
- 공유 루트/다이어리 공통 reaction 모델 검토

### Step 7. i18n 준비

목표: 일본 사용자 시나리오를 위해 한국어/영어/일본어 확장이 가능한 구조를 만든다.

- 하드코딩 문구를 점진적으로 dictionary로 이동
- `AppLanguage`를 `ko | en | ja`로 확장 준비
- 코스/스팟/POI 이름도 다국어 필드 설계
